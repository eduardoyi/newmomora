# Cloudflare illustration Workflow deployment

Memory illustration generation is split between Supabase and Cloudflare
Workflows. Supabase authenticates callers, authorizes the family, performs
emotion/safety work, owns job/publication state, and writes the final memory
row. Cloudflare receives a deterministic job ID, obtains temporary job input
through an HMAC-protected bridge, calls OpenAI, and transfers generated bytes
to the existing private R2 bucket.

For the design rationale, review findings, production lessons, and the
portrait-generation migration blueprint, read
[Durable AI generation workflow playbook](./durable-ai-generation-workflows.md).

## Secrets and configuration

Do not place any of these in the Expo bundle or `.env.example`:

- Supabase `MEMORY_ILLUSTRATION_BACKEND=legacy|cloudflare` is server-only and
  defaults to `legacy` until cutover.
- The Cloudflare Worker holds its own `OPENAI_API_KEY`, R2/Images bindings,
  bridge HMAC secret, and Supabase-to-Cloudflare dispatch secret.
- The Supabase bridge function runs with JWT verification disabled, but
  verifies a timestamped raw-body HMAC before parsing or reading any job.
  It has no browser/client route.
- Cloudflare does not receive the Supabase service-role key. Job rows have
  RLS with no client policies; authorized Supabase server code owns them.

## Production rollout and rollback

Use Node 22 or newer for the Worker. From
`cloudflare/memory-illustration-worker`, run `nvm use` (the checked-in
`.nvmrc` selects 22) before any Wrangler command. The `staging` configuration
contains deliberate placeholders; never use it for a production deployment.

1. Ship Supabase first, leaving the old path active:

   ```sh
   supabase db push
   supabase functions deploy generate-illustration
   supabase functions deploy generate-portrait-illustration
   supabase functions deploy workflow-illustration-bridge
   supabase functions deploy delete-storage-object
   supabase secrets set MEMORY_ILLUSTRATION_BACKEND=legacy
   ```

   This deploys the job schema, the shared `gpt-image-1.5` portrait fallback,
   HMAC bridge, and the storage-key parser used to clean up immutable
   `/illustrations/{generationId}.webp` outputs while the existing Edge
   Function still generates memory illustrations. Confirm a legacy
   illustration claim writes
   `illustration_generation_started_at` before continuing.

2. Configure the Worker from its own directory. Substitute values only at the
   interactive prompts; do not add them to `wrangler.jsonc`, `.dev.vars.example`,
   or source control:

   Before the production deploy, confirm the Momora Cloudflare account is
   still on the Workers Paid plan selected for higher production capacity.
   Cloudflare currently offers Workflows on both Free and Paid plans, so this
   is Momora's operational choice—not a blanket platform prerequisite. Local
   Worker tests and dry-runs do not require a plan change. See the dated cost
   record in
   [the durable-generation playbook](./durable-ai-generation-workflows.md#cost-decision-record-2026-07-22).

   ```sh
   cd cloudflare/memory-illustration-worker
   nvm use
   npm ci
   npx wrangler secret put OPENAI_API_KEY --env=""
   npx wrangler secret put DISPATCH_SIGNING_SECRET --env=""
   npx wrangler secret put SUPABASE_BRIDGE_HMAC_SECRET --env=""
   npm run deploy:dry-run
   npx wrangler deploy --env=""
   ```

   `--env=""` explicitly targets the top-level production configuration.
   It must be present for dry-runs and deploys so Wrangler cannot silently
   select the placeholder staging environment.

3. Wire Supabase to that deployed Worker. Set `MOMORA_WORKER_DISPATCH_URL`,
   `MOMORA_DISPATCH_SECRET`, and `MOMORA_BRIDGE_SECRET` only in the current
   secure shell session (or use the Supabase secrets UI); do not put their
   values in a command history, source file, or tracked environment file.
   The URL variable includes the Worker `/dispatch` path. Then preserve the
   two exact pairings:

   ```sh
   supabase secrets set CLOUDFLARE_ILLUSTRATION_WORKFLOW_URL="$MOMORA_WORKER_DISPATCH_URL"
   supabase secrets set CLOUDFLARE_ILLUSTRATION_DISPATCH_SECRET="$MOMORA_DISPATCH_SECRET"
   supabase secrets set CLOUDFLARE_ILLUSTRATION_BRIDGE_SECRET="$MOMORA_BRIDGE_SECRET"
   ```

   Configure the Worker with its own `OPENAI_API_KEY` secret value. Supabase
   retains its separate OpenAI secret for safety/emotion/chat, portraits, and
   the legacy rollback path. The Worker never receives a Supabase service-role
   key. The bridge function is already configured with `verify_jwt = false`;
   its HMAC remains mandatory.

4. Release the timestamp/requestIntent-aware mobile client while
   `MEMORY_ILLUSTRATION_BACKEND=legacy`. Installed older clients remain safe,
   but their Regenerate control can be enabled at three minutes and reuse an
   active job until the five-minute-thirty-second recovery boundary. Allow the
   updated client release to propagate before backend cutover.

5. Smoke-test while `MEMORY_ILLUSTRATION_BACKEND=legacy`: use a synthetic
   illustrated memory in the app, confirm the legacy result still publishes,
   and confirm the Worker health endpoint responds. Then set
   `MEMORY_ILLUSTRATION_BACKEND=cloudflare`, create one synthetic illustrated
   memory, and verify one Workflow instance, one job terminal state, a ready
   memory row, and the expected private R2 key. Inspect only IDs, status codes,
   model names, and durations—not prompts, names, content, reference keys, or
   image bytes.

6. Monitor dispatch acceptance, instance-idempotency, provider attempts,
   terminal failures, publication CAS losses, generation duration, and retry
   counts. To roll back, set `MEMORY_ILLUSTRATION_BACKEND=legacy`. Running
   Workflow jobs remain safe because Supabase publication matches their
   attempt ID.

The Workflow uses a five-minute lease: 4:30 for provider work and 30 seconds
for finalization. Client automatic recovery waits 5:30 for `generating` rows;
`pending` remains recoverable after three minutes, including rows whose app
was terminated before dispatch. There is no R2 staging bucket, lifecycle rule,
or scheduled orphan sweep in the current single-user phase.
