# Cloudflare image-generation Workflow deployment

Memory illustration and portrait generation are split between Supabase and the
same Cloudflare Worker/Workflows deployment. Supabase authenticates callers,
authorizes the family, owns job/publication state, and writes the final public
row. Cloudflare receives a deterministic job ID, obtains temporary job input
through an HMAC-protected bridge, calls OpenAI, and transfers generated bytes
to the existing private R2 bucket.

Generated-output writes use a database upload lease for both pipelines. The
Worker authorizes the exact job/key immediately before PUT, performs only
bounded same-key/same-byte R2 retries under that UUID token, and records the
exact token after definite success. A thrown or `null` PUT result, or an
ambiguous completion response, leaves the lease intact and skips terminal
failure/cleanup. Deterministic HEAD recovery must authorize and complete its
token before publish; it never bypasses an active/unconfirmed upload.

For the design rationale, review findings, production lessons, and the
portrait-generation migration blueprint, read
[Durable AI generation workflow playbook](./durable-ai-generation-workflows.md).

## Secrets and configuration

Do not place any of these in the Expo bundle or `.env.example`:

- Supabase `MEMORY_ILLUSTRATION_BACKEND=legacy|cloudflare` and
  `PORTRAIT_GENERATION_BACKEND=legacy|cloudflare` are independent,
  server-only flags. Both default to `legacy` until their cutovers.
- The Cloudflare Worker holds its own `OPENAI_API_KEY`, R2/Images bindings,
  memory and portrait bridge HMAC secrets, and memory and portrait
  Supabase-to-Cloudflare dispatch secrets.
- The Supabase bridge function runs with JWT verification disabled, but
  verifies a timestamped raw-body HMAC before parsing or reading any job.
  It has no browser/client route.
- Cloudflare does not receive the Supabase service-role key. Job rows have
  RLS with no client policies; authorized Supabase server code owns them.
- Portrait completion asks the ordinary memory dispatcher to reconsider up to
  three deferred memories through a separate Supabase-only timestamp HMAC.
  It does not forward the initiating user's JWT through a Workflow.

## Production rollout and rollback

Use Node 22 or newer for the Worker. From
`cloudflare/memory-illustration-worker`, run `nvm use` (the checked-in
`.nvmrc` selects 22) before any Wrangler command. The `staging` configuration
contains deliberate placeholders; never use it for a production deployment.

1. Drain the already-deployed memory Workflow before changing its database
   contract. Flip memory generation to legacy first; keep portrait legacy too:

   ```sh
   supabase secrets set MEMORY_ILLUSTRATION_BACKEND=legacy
   supabase secrets set PORTRAIT_GENERATION_BACKEND=legacy
   ```

   Wait at least 5 minutes 30 seconds after the secret update, then query the
   production database and require zero rows before proceeding:

   ```sql
   select id, status, started_at
   from public.memory_illustration_jobs
   where status in ('queued', 'running');
   ```

   A non-empty result stops the rollout; do not apply the stricter
   upload-completion publication contract while an old Worker instance can
   still publish. If this is a resumed/partial rollout and the jobs table
   already has `upload_token`, also require zero rows where `upload_token is
   not null`. Do not clear either state by hand merely to pass the check;
   reconcile/recover the job first.

   Production currently has one account in the 15-day legacy deletion grace
   period. Before the migration, record its `id`, `deleted_at`, and
   `scheduled_hard_delete_at` from the following PII-minimized query so the
   same row can be compared after deployment. Require exactly one row and stop
   if it is already due during the rollout window:

   ```sql
   select id, deleted_at, scheduled_hard_delete_at
   from public.user_profiles
   where scheduled_hard_delete_at is not null;
   ```

2. Apply the database and Supabase function changes while both generators
   remain on legacy:

   ```sh
   supabase db push
   supabase functions deploy generate-illustration
   supabase functions deploy generate-portrait-illustration
   supabase functions deploy workflow-illustration-bridge
   supabase functions deploy workflow-portrait-bridge
   supabase functions deploy delete-family-member
   supabase functions deploy delete-portrait-version
   supabase functions deploy delete-user-account
   supabase functions deploy cancel-account-deletion
   supabase functions deploy hard-delete-expired-accounts
   supabase functions deploy delete-storage-object
   ```

   This deploys the memory and portrait job schemas, bridge functions, and
   storage-key parsers while both legacy paths remain active. Confirm that a
   legacy memory claim writes `illustration_generation_started_at` and a
   legacy portrait claim writes `generation_started_at` before continuing.
   After the migration, require zero rows in
   `memory_illustration_jobs` where `upload_token is not null`; this catches a
   partial prior rollout before the new Worker becomes reachable:

   ```sql
   select id, status, upload_started_at
   from public.memory_illustration_jobs
   where upload_token is not null;
   ```

   Verify the legacy scheduled-deletion row survived the migration unchanged,
   received its operation token, is not accidentally claimed for hard delete,
   and any owned family soft-deleted by that same scheduling transition carries
   the exact same token. Compare the ID and two timestamps with the preflight
   snapshot and require every boolean below to be `true`:

   ```sql
   select
     profile.id,
     profile.deleted_at,
     profile.scheduled_hard_delete_at,
     profile.account_deletion_token is not null as has_operation_token,
     profile.hard_delete_token is null as is_not_hard_delete_claimed,
     not exists (
       select 1
       from public.families family
       where family.owner_id = profile.id
         and family.deleted_at between profile.deleted_at - interval '1 minute'
           and profile.deleted_at + interval '1 minute'
         and family.account_deletion_token is distinct from profile.account_deletion_token
     ) as owned_family_tokens_match
   from public.user_profiles profile
   where profile.scheduled_hard_delete_at is not null;
   ```

   Also confirm the Supabase deployment list shows the new versions of
   `delete-user-account`, `cancel-account-deletion`, and
   `hard-delete-expired-accounts`, and that the existing cron remains enabled:

   ```sql
   select jobname, active
   from cron.job
   where jobname = 'invoke-hard-delete-expired-accounts';
   ```

   Do not exercise cancel or hard delete against this real grace-period row as
   a smoke test. Its unchanged timestamps, token backfill/family-token match,
   active cron, and the shipped Edge/SQL test suite are the compatibility
   proof; destructive behavior must use synthetic fixtures only.

3. Configure the Worker from its own directory. Substitute values only at the
   interactive prompts; do not add them to `wrangler.jsonc`, `.dev.vars.example`,
   or source control:

   Before the production deploy, confirm the Momora Cloudflare account is
   still on the Workers Paid plan selected for higher production capacity.
   Cloudflare currently offers Workflows on both Free and Paid plans, so this
   is Momora's operational choice—not a blanket platform prerequisite. Local
   Worker tests and dry-runs do not require a plan change. See the dated cost
   record in
   [the durable-generation playbook](./durable-ai-generation-workflows.md#cost-decision-record-2026-07-22).

   ```sh
   cd cloudflare/memory-illustration-worker
   nvm use
   npm ci
   npx wrangler secret put OPENAI_API_KEY --env=""
   npx wrangler secret put DISPATCH_SIGNING_SECRET --env=""
   npx wrangler secret put SUPABASE_BRIDGE_HMAC_SECRET --env=""
   npx wrangler secret put PORTRAIT_DISPATCH_SIGNING_SECRET --env=""
   npx wrangler secret put PORTRAIT_SUPABASE_BRIDGE_HMAC_SECRET --env=""
   npm run deploy:dry-run
   npx wrangler deploy --env=""
   ```

   `--env=""` explicitly targets the top-level production configuration.
   It must be present for dry-runs and deploys so Wrangler cannot silently
   select the placeholder staging environment.

4. Wire Supabase to that deployed Worker. Set memory and portrait dispatch/
   bridge values only in the current
   secure shell session (or use the Supabase secrets UI); do not put their
   values in a command history, source file, or tracked environment file.
   The URL variable includes the Worker `/dispatch` path. Then preserve the
   two exact pairings:

   ```sh
   supabase secrets set CLOUDFLARE_ILLUSTRATION_WORKFLOW_URL="$MOMORA_WORKER_DISPATCH_URL"
   supabase secrets set CLOUDFLARE_ILLUSTRATION_DISPATCH_SECRET="$MOMORA_DISPATCH_SECRET"
   supabase secrets set CLOUDFLARE_ILLUSTRATION_BRIDGE_SECRET="$MOMORA_BRIDGE_SECRET"
   supabase secrets set CLOUDFLARE_PORTRAIT_WORKFLOW_URL="$MOMORA_PORTRAIT_WORKER_DISPATCH_URL"
   supabase secrets set CLOUDFLARE_PORTRAIT_DISPATCH_SECRET="$MOMORA_PORTRAIT_DISPATCH_SECRET"
   supabase secrets set CLOUDFLARE_PORTRAIT_BRIDGE_SECRET="$MOMORA_PORTRAIT_BRIDGE_SECRET"
   supabase secrets set PORTRAIT_MEMORY_RETRIGGER_SECRET="$MOMORA_PORTRAIT_MEMORY_RETRIGGER_SECRET"
   ```

   Configure the Worker with its own `OPENAI_API_KEY` secret value. Supabase
   retains its separate OpenAI secret for safety/emotion/chat, portraits, and
   the legacy rollback path. The Worker never receives a Supabase service-role
   key. The bridge function is already configured with `verify_jwt = false`;
   its HMAC remains mandatory.

5. Release the timestamp/requestIntent-aware mobile client while both flags
   remain `legacy`. The portrait client recovers unclaimed pending rows at
   three minutes and claimed work at five minutes thirty seconds without
   writing state; it disables fresh active regeneration. Allow the updated
   client release to propagate before portrait backend cutover.

6. Smoke-test while `MEMORY_ILLUSTRATION_BACKEND=legacy`: use a synthetic
   illustrated memory in the app, confirm the legacy result still publishes,
   and confirm the newly deployed Worker health endpoint responds. Only after
   the new bridge and Worker are both live, set
   `MEMORY_ILLUSTRATION_BACKEND=cloudflare`, create one synthetic illustrated
   memory, and verify one Workflow instance, one job terminal state, a ready
   memory row, and the expected private R2 key. Keep
   `PORTRAIT_GENERATION_BACKEND=legacy` throughout this memory smoke and until
   the separate portrait cutover proof below. Inspect only IDs, status codes,
   model names, and durations—not prompts, names, content, reference keys, or
   image bytes.

7. Monitor dispatch acceptance, instance-idempotency, provider attempts,
   upload authorization/completion, active upload leases, terminal failures,
   publication CAS losses, generation duration, and retry counts. To roll
   back, set `MEMORY_ILLUSTRATION_BACKEND=legacy`. Running Workflow jobs remain
   safe because Supabase publication matches their attempt ID and requires a
   confirmed upload token.

The illustration Workflow uses a five-minute lease with 30 seconds reserved
for finalization. The portrait Workflow uses the same total lease, with one
bounded primary attempt (up to 180 seconds) and one retryable fallback (up to
60 seconds) when enough time remains. Client automatic recovery waits 5:30
for claimed rows; unclaimed `pending` remains recoverable after three minutes,
including rows whose app was terminated before dispatch. There is no R2 staging
bucket, lifecycle rule, or scheduled orphan sweep in the current single-user
phase.

Before either cutover, run the upload-race suite as well as the ordinary happy
path. It must prove: a consumed provider reservation makes no paid call; a
HEAD hit completes an active or fresh no-op lease before publish; authorization
rejection prevents PUT/publish; an object appearing after the first HEAD is
not overwritten; R2 exceptions and `null` results never clear the lease; and
ambiguous completion never invokes failure cleanup. Entity/account deletion
and stale recovery must not reclaim a job while its upload lease is fresh.

## Portrait cutover proof

Keep `PORTRAIT_GENERATION_BACKEND=legacy` until the Worker deployment and
client recovery release are live. Create one synthetic fixture-only family
member/version under the authorized production test account; never use an
organic child photo. Enable `PORTRAIT_GENERATION_BACKEND=cloudflare`, verify
the portrait Workflow/job/claim state, private WebP object and signed display,
terminal private-input scrubbing, retained old portrait on regeneration, and a
deferred-memory retrigger. Inspect only IDs, statuses, model names, and
durations. Delete every synthetic database row and R2 key afterward, confirm
the test prefix is empty, then observe one ordinary portrait before removing
the legacy path. Roll back immediately by resetting only
`PORTRAIT_GENERATION_BACKEND=legacy`; an in-flight Workflow cannot publish
unless its attempt token still owns the row.
